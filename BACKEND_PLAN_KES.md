# BACKEND_PLAN.md — оживление мокапа в боевую CRM

Этот файл — пошаговый план превращения статического мокапа (данные в `data.js` +
localStorage) в боевую CRM с базой данных, API и авторизацией на стеке Cloudflare.

Claude Code: выполняй слои **строго по порядку**, по одному за раз.
**Не переходи к следующему слою, пока текущий не завершён и не проверен.**
После каждого слоя выводи краткий отчёт и **жди подтверждения**, прежде чем продолжать.

---

## Контекст проекта

- **Проект:** KES CRM — CRM для ТОО KazEnergoSnab, дистрибьютор электротехники EKF
  в Караганде. Каталог 1100+ SKU, склад, отгрузки, B2B-сделки, дилеры.
- **Стек:** vanilla JS, без фреймворков, без TypeScript, без новых npm-зависимостей
  (кроме самого `wrangler` для деплоя). Бэкенд — Cloudflare Workers.
- **Хранение:** Cloudflare D1 (база), R2 (файлы).
- **Текущее состояние:** данные захардкожены в `data.js` (объект `SEED`), приложение
  читает/пишет в localStorage браузера (ключ `kes_crm_state_v3`). Настоящего бэкенда,
  базы и авторизации нет. «Логин» сейчас — сравнение с паролем `'demo'` в коде.
- **Цель:** общая база на сервере, реальный вход, данные сохраняются и видны всем
  пользователям, не пропадают и не разделены по устройствам.

### Главное правило (критично для будущей передачи клиенту)

Этот проект — продукт, который позже передаётся клиенту на его собственный
Cloudflare-аккаунт. Поэтому:

- **Никаких хардкод-строк конкретного клиента в коде.** Название компании,
  стадии воронки, источники, склады, категории товаров, типы полей — всё это данные
  в БД, а не значения в коде. Тот же код должен работать у любого клиента без правок.
- **Все ID и имена ресурсов — только в `wrangler.toml`**, не разбросаны по коду.
  При передаче клиент меняет один файл на свои ресурсы.
- **Схема БД — отдельный файл `schema.sql`**, который накатывается на любую новую
  базу одной командой.

## Слой 1 — Схема базы данных

**Цель:** спроектировать структуру БД на основе данных из `data.js`.

1. Изучи `data.js` (объект `SEED`) и разбери все сущности: компания, пользователи,
   клиенты (B2B-контрагенты, дилеры), сделки, каталог товаров/SKU (1100+ позиций),
   склад/остатки, отгрузки, и все справочники (стадии сделок, категории товаров,
   типы клиентов/дилеров, источники, склады, статусы отгрузок).
2. Спроектируй реляционную схему для D1 (SQLite-совместимую): таблицы, поля, типы,
   первичные и внешние ключи, индексы где нужно. Учти большой каталог (1100+ SKU) —
   таблица товаров должна быть проиндексирована для поиска.
3. Справочники (стадии, категории, типы, склады, статусы) — выноси в отдельные
   таблицы, НЕ оставляй значениями в коде. Это требование клиент-независимости.
4. Замени демо-вход: добавь таблицу `users` для настоящей авторизации — email/логин,
   хэш пароля (НЕ открытый текст!), роль. Роли возьми из мокапа.
5. Создай файл `schema.sql` в корне репозитория со всеми `CREATE TABLE` и индексами.
6. **НЕ применяй схему к базе.** Только создай файл.

**Отчёт по слою:** перечисли простым языком, какие таблицы получились и как они
связаны. Жди подтверждения.

## Слой 2 — Worker + API (CRUD)

**Цель:** серверный код, который обслуживает запросы фронтенда и работает с D1/R2.

1. Создай `wrangler.toml` с привязками к существующим ресурсам (значения выше).
   Имена биндингов выбери короткие и универсальные (например `DB` для D1,
   `BUCKET` для R2), чтобы код не был привязан к имени конкретного клиента.
2. Создай Worker (точка входа, например `worker/index.js`, или Pages Functions
   в папке `functions/` — выбери подход, совместимый с тем, что проект уже на
   Cloudflare Pages; опиши выбор в отчёте).
3. Реализуй REST API: для каждой основной сущности — эндпоинты
   создания/чтения/обновления/удаления (CRUD), плюс эндпоинты для справочников.
   Для каталога товаров — эндпоинт поиска/фильтрации с пагинацией (1100+ SKU).
4. Реализуй загрузку файлов в R2 (если в мокапе есть вложения/изображения товаров).
5. Все ответы — JSON. Обрабатывай ошибки, возвращай понятные коды.
6. НЕ трогай пока фронтенд — только серверная часть.

**Отчёт по слою:** список созданных эндпоинтов (метод + путь + назначение).
Жди подтверждения.

## Слой 3 — Авторизация (JWT)

**Цель:** заменить демо-вход (пароль `'demo'` в коде) на настоящую авторизацию.

1. Реализуй эндпоинт `/api/login`: принимает логин/пароль, сверяет с хэшем
   в таблице `users`, выдаёт JWT.
2. Защити API-эндпоинты: проверка JWT на запросах, требующих авторизации.
3. Хэширование паролей — стойкое (не простой хэш). Секрет для подписи JWT —
   через секрет wrangler, НЕ в коде.
4. Предусмотри роли из мокапа и проверку прав где это уместно.

**Отчёт по слою:** как устроен вход, какие эндпоинты защищены. Жди подтверждения.

## Слой 4 — Фронтенд: localStorage → API

**Цель:** переключить интерфейс с localStorage на реальные API-вызовы.
Это самый объёмный слой — действуй аккуратно, по частям.

1. Найди все места, где код читает/пишет localStorage (ключ `kes_crm_state_v3`).
2. Замени их на `fetch` к API из Слоя 2. Сохраняй JWT (из Слоя 3) и подставляй
   его в запросы.
3. Экран входа подключи к `/api/login` вместо сравнения с паролем `'demo'`.
4. Сохрани прежнее поведение и вид интерфейса — мокап утверждён, UI менять НЕ нужно,
   меняется только источник данных.
5. Делай по разделам (например: сначала каталог товаров, потом клиенты, сделки,
   склад, отгрузки), чтобы можно было тестировать частями.

**Отчёт по слою:** какие разделы переключены на API, что проверено.
Жди подтверждения.

## Слой 5 — Начальные данные, деплой, проверка

**Цель:** запустить всё вживую и убедиться, что данные сохраняются и общие.

1. Подготовь сидинг: перенеси нужные начальные/справочные данные из `data.js`
   в SQL-вставки (`seed.sql`). Разделяй: справочники (нужны всегда), каталог
   товаров (если переносим демо-каталог), демо-записи (опционально).
2. Опиши команды применения схемы и сидов к базе `kes-crm-db`
   (пользователь выполнит их сам — НЕ выполняй их без подтверждения):
   - применить `schema.sql` к D1;
   - применить `seed.sql` к D1;
   - задеплоить Worker.
3. Опиши, как проверить: вход работает, данные сохраняются, видны после
   перезагрузки и с другого устройства/браузера.

**Отчёт по слою:** инструкция запуска и чек-лист проверки. Жди подтверждения.
